# macOS 26 Liquid Glass UI/UX 升级方案：Sidebar, Header & Input
**Author: Alan Dye (Apple VP of Human Interface Design)**
**Date: 2026-02-22**
**Scope: AgentOrientedTUI / host GUI**

---

你好。我仔细看了你的线框图和需求。你对功能重组的直觉非常敏锐——将 Agent 控制区与输入框结合，以及精简 Header，这些都是让界面更符合逻辑的正确方向。

但在 macOS 26 的 Liquid Glass 设计哲学下，我们需要对空间布局（Spatial Layout）和视觉层级（Visual Hierarchy）进行更严谨的推敲。以下是我的设计方案与专业建议：

## 1. Sidebar：Session 卡片的呼吸感与克制

你的方向完全正确，我们需要更多的元数据（时间）和操作项（更多菜单），但要以“克制”的方式呈现。

### 设计方案
*   **时间戳 (Time Ago)**：
    *   **UI 表现**：使用 `SF Pro Text`，字号 `11px` (Caption)，颜色使用 `Tertiary Label` (低对比度)。
    *   **位置**：放在卡片内部的左下角或标题右侧，与标题形成清晰的主次关系。
*   **更多操作按钮 (⋯)**：
    *   **Alan 的建议（克制设计）**：不要让每个卡片默认都显示这个按钮，这会造成极大的视觉噪音（Visual Clutter）。
    *   **交互逻辑**：默认隐藏。只有当鼠标 **Hover** 到该 Session 卡片时，右上角才平滑浮现 `⋯` 按钮（必须使用 macOS 经典的带圈省略号图标，即 SF Symbol `ellipsis.circle`）。
    *   **菜单呈现**：点击后弹出的菜单必须是标准的 macOS 26 Liquid Glass Popover（带毛玻璃背景、圆角、阴影），包含：`Pin (固定)`、`Rename (重命名)`、`Delete (删除，标红)`。

## 2. Header：交通灯与汉堡菜单的“空间博弈”

**⚠️ Alan 的设计纠偏：关于汉堡菜单的位置**

你提到：“*Title 浮岛下方展示 Hamburger 按钮... 因为会被 macOS 的 traffic light (红黄绿灯) 挡住*”。
**这是一个典型的“为了解决局部冲突而破坏全局结构”的妥协设计。** 在 macOS 中，把核心导航按钮悬挂在标题下方，会破坏顶部的视觉平衡，产生一种“掉落”的破碎感。

### 真正的 macOS 26 解决方案：动态响应的 Toolbar
我们需要让 Header 成为一个智能的 Liquid Glass 浮岛群，优雅地处理交通灯的占位。

*   **左侧：Window Controls & Sidebar Toggle**
    *   **当 Sidebar 展开时**：交通灯 (Traffic Lights) 位于 Sidebar 的左上角。此时，Hamburger 按钮可以放在 Sidebar 的右上角，或者 Header 浮岛的最左侧。
    *   **当 Sidebar 收起时**：交通灯会自动回到主窗口的左上角。此时，Hamburger 按钮应该**紧贴在交通灯的右侧**，与交通灯处于同一水平线上。
    *   **UI 表现**：Hamburger 按钮应该是一个精致的 Liquid Glass 材质的方形/圆形图标按钮（Icon Button），而不是孤零零地飘在标题下面。
*   **中心：Title 浮岛**
    *   **UI 表现**：居中对齐（或在左侧控件右边保持一定间距）。它是一个独立的 Liquid Glass 胶囊，只展示当前 Session 的标题。
*   **右侧：View Mode 切换浮岛**
    *   **UI 表现**：如你所说，与 Title 浮岛顶部对齐，右边界与下方内容区对齐。使用 Segmented Control（分段控制器）样式，材质同样为 Liquid Glass。

**Header 最终形态**：
```text
[🔴🟡🟢] [≡]        [   Session Title   ]                 [ Chat | TUI View ]
```
*(所有元素都在同一水平高度，作为独立的 Liquid Glass 岛屿悬浮在背景之上)*

## 3. 输入框与 Agent 控制区：对话的上下文闭环

你提出将 Session 控制（Idle/Pause）移到输入框左侧，并删除 Delete 按钮。**这是极其出色的 UX 决策。**
Agent 的状态（思考中/空闲）和用户的输入，本质上是“对话”的两个端点，它们在物理空间上靠近，能极大地增强用户的掌控感。Delete 属于会话管理，确实应该只留在 Sidebar。

### 设计方案
*   **整体布局**：底部不再是一个贯穿全宽的底栏，而是悬浮在内容上方的**控制台岛屿 (Console Island)**。
*   **左侧：Agent 状态胶囊**
    *   **UI 表现**：一个小巧的 Liquid Glass 胶囊。
    *   **内容**：状态指示灯（如绿点代表 Idle，呼吸的紫点代表 Thinking） + 状态文本 + 暂停/恢复按钮 (⏸ / ▶)。
*   **右侧：用户输入框 (User Command)**
    *   **宽度重构**：摒弃全宽设计。输入框应该有 `max-width`（例如 800px），并在可用空间内居中，或者与上方的聊天气泡宽度对齐。这能大幅提升长文本的阅读舒适度。
    *   **UI 表现**：一个大圆角的 Liquid Glass 文本域。当用户 Focus 时，边缘产生微妙的系统蓝折射高光（Refraction Highlight），而不是粗暴的实线边框。
    *   **发送按钮**：内嵌在输入框右侧，当输入框有内容时，图标变为系统强调色（Accent Color）。

**底部控制区最终形态**：
```text
      [ 🟢 Idle | ⏸ ]  [ Enter command...                                ⇧ ]
```

## 5. 附加建议：Chat Bubbles (聊天气泡)

从你的线框图中我注意到，User Command 的气泡非常巨大且宽。
在 macOS 26 的排版规范中，内容应该具有良好的“呼吸感”。
*   **最大宽度**：聊天气泡不应超过容器宽度的 75% - 80%。
*   **材质区分**：
    *   **User Message**：使用系统强调色（如蓝色）的半透明变体（Liquid Glass Clear），文字为白色。
    *   **Agent Message**：使用 Standard Material（如 `rgba(255,255,255,0.05)`），不使用强烈的背景模糊，让它安静地坐在内容层。

## 总结

你的重构思路非常棒，我们只需要用 Apple 的设计语言将其“翻译”出来：
1. **Sidebar**：增加时间戳，Hover 时显示 `⋯` 菜单（保持克制）。
2. **Header**：拒绝将汉堡菜单下放。让交通灯、汉堡菜单、标题、视图切换在**同一水平线**上形成错落有致的 Liquid Glass 浮岛。
3. **Input Area**：将 Agent 状态与输入框组合成底部的悬浮控制台，限制输入框最大宽度，提升专注度。
