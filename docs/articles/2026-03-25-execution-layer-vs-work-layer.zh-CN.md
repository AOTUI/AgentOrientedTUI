# 本地、容器、VM 都重要，但 Agent 时代真正缺的可能不是执行层

我现在越来越觉得，行业对 Agent 的讨论有点跑偏了。

大家一边在卷模型，一边在卷执行环境：本地、容器、VM、microVM、devcontainer、沙箱隔离……好像下一阶段的胜负，主要就看 Agent 跑在哪里。

但很多时候，这场争论争的其实只是“壳”。

壳当然重要。

可问题是，壳再硬、隔离再强、环境再标准化，也不等于 Agent 就有了一个真正适合工作的环境。

因为执行层决定的是 Agent **在哪儿跑**。  
真正更影响上限的，往往是 Agent **在什么里面工作**。

这不只是效率问题，也是安全问题。

一旦 Agent 真开始拿到电脑控制权、浏览器控制权、代码执行权，问题就不只是“权限给多大”，而是：

**你到底让它在一个什么样的边界里工作？**

如果这一层没想清楚，执行环境再豪华，也只是把 Agent 放进了一个更安全的壳里，而不是一个更好的工作环境里。

所以我的判断是：

**本地、容器、VM 当然重要。**  
**但 Agent 时代真正缺的，可能不是执行层，而是工作层。**

---

## 先把话说死一点：本地、容器、VM 不是替代关系

我不觉得本地、容器、VM 最后会收敛成一个唯一赢家。

它们更像是 Agent 时代执行层的三种基本形态，而且大概率会长期共存、长期分工。

原因很简单：  
它们解决的，本来就不是同一类问题。

### 本地

本地的优势很直接：

- 最贴近用户真实工作环境
- 延迟最低
- 跟文件系统、IDE、浏览器、桌面协作最自然
- 最适合强交互、人机共驾、个人开发者场景

但它的问题也很直接：

- 权限太真实
- 环境太脏
- 隔离最难
- 一旦 Agent 能力变强，误操作和越权风险也最直接

所以本地更像什么？

更像“Agent 和人并肩工作”的场景。  
它天然适合协作，不天然适合让 Agent 在一个完全放开的环境里自己野跑。

### 容器

容器为什么会成为现在最主流的一条路？

因为它在工程上确实很顺：

- 环境一致性强
- 依赖好管理
- 成本相对低
- 编排方便
- 很适合大规模任务分发

对开发、测试、网页操作、批处理场景来说，容器已经非常好用了。

但容器也不是万能解。

它能解决很多问题，但不是所有问题都适合放在容器这一层解决。  
一旦进入高风险操作、多租户治理、强安全诉求场景，容器未必就是最终答案。

所以容器更像 Agent 时代执行层里的主力工作马。  
很重要，但不包打天下。

### VM / microVM

VM，尤其是 microVM，解决的是另一类问题：

- 隔离要更硬
- 会话边界要更清晰
- 高风险任务要更可控
- 多租户环境里要更容易做治理和审计

它的价值不在“更方便”，而在“更稳、更硬、更容易建立强边界”。

代价也很明显：

- 成本更高
- 调度更复杂
- 冷启动、调试、开发体验通常更重

所以 VM / microVM 很可能会成为很多高风险、高价值、高治理要求场景下的核心执行层。

但它也不可能把本地和容器都替代掉。

### 这一层怎么理解最合理

如果一定要用一句话概括，我会这么说：

- 本地负责贴身协作
- 容器负责大规模工程执行
- VM / microVM 负责强隔离和高风险任务

它们不太像谁消灭谁。  
更像谁负责哪类任务、哪类风险、哪类协作方式。

如果只回答到这里，其实已经能解释很多关于“执行层主战场”的讨论了。

但这不是我最想说的部分。

因为这只回答了问题的一半。

---

## 真正更缺的，不是“跑在哪儿”，而是“在什么里面工作”

这是我觉得最关键、也最容易被忽视的一点。

就算我们把本地、容器、VM 哪类场景更适合讲得很清楚，问题也只解决了一半。

因为这些技术回答的主要还是：

**Agent 在哪儿执行。**

它们没有自动回答另一个更深的问题：

**Agent 到底在什么样的环境里工作。**

这两个问题看起来很像，实际上完全不是一回事。

执行层主要解决的是这些事：

- 进程在哪儿跑
- 权限怎么隔离
- 环境怎么初始化
- 资源怎么分配
- 会话怎么清理
- 风险边界怎么划

这些都很重要。

但它不自动解决这些问题：

- 状态怎么呈现给 Agent
- Agent 眼里的“当前世界”到底长什么样
- 动作和状态怎么绑定
- 工作对象怎么保活、更新、失效、移除
- 长任务怎么避免上下文越来越脏

也就是说：

你可以给 Agent 一个很安全的容器。  
你也可以给它一个隔离很强的 microVM。

但这不等于你已经给了它一个适合工作的环境。

很多系统做到最后，其实只是把 Agent 放进了一个边界更硬的执行壳里。

壳当然更安全。  
但它是不是一个好的工作环境，这是另一回事。

而我觉得，现在行业真正缺的，就是这一层。

如果一定要先给它起个名字，我会把它叫做：

**工作层。**

后面我会把它具体展开成一个更明确的东西：**Agent Apps**。

---

## 真正棘手的，不只是权限大，而是工作边界太差

所以我一直觉得，安全问题不能只从“权限大小”来理解。

比如大家讨论 Claude 获得电脑控制权，或者各种 computer use、browser use、desktop use 的时候，第一反应通常都是这些：

- 它能不能删文件
- 它能不能执行危险命令
- 它会不会误操作系统
- 它会不会越权访问
- 它拿到系统权限之后会不会失控

这些问题当然都对。

但如果只停在这里，还是太表面。

因为真正危险的，不只是“权限大”。

而是：

**我们是不是默认让 Agent 直接面对了一个过于宽、过于原始、过于缺少工作结构的环境。**

这个区别很重要。

如果 Agent 直接面对的是整台电脑，那么它面对的其实是一个自由度极高、语义层次极低、边界极不清晰的整体：

- 文件系统
- 浏览器
- shell
- GUI
- 系统设置
- 各种应用
- 各种临时状态
- 各种历史残留

这时候问题就不只是：

> 要不要再少给一点权限？

而是：

> 你是不是一开始就把它扔进了一个不该直接面对的环境里？

所以我越来越认同一个判断：

**安全的关键，不只是减少权限，而是减少“无结构的自由度”。**

这句话很重要。

因为很多人一谈安全，直觉就是“权限收紧”。  
但 Agent 系统里更根本的事，往往不是把能力一刀砍掉。

而是：

- 让它工作的环境更清晰
- 让它能操作的对象更明确
- 让它面对的状态更收束
- 让它的动作只在一个有语义的边界里发生

权限太大当然危险。

但环境太原始、太宽、太散，也一样危险。  
而且这种危险更隐蔽。

因为它不只是安全风险，还是效率风险、一致性风险和可靠性风险。

---

## Agent Apps 真正补的，不是功能层，而是工作层

这时候，Agent Apps 才开始真正变得重要。

但我想先强调一点：

**Agent Apps 不是“给 Agent 准备的一些工具壳子”。**

它也不是“把现有软件换个 AI-friendly UI”这么简单。

如果只是这么理解，就把它看浅了。

我更愿意把 Agent Apps 理解成：

**给 Agent 准备的一层工作边界。**

我们之所以长期在 IDE、浏览器、文档编辑器、设计工具、任务管理工具里工作，不只是因为这些 app 更顺手。

更重要的是，app 本身就是一种边界。

它同时做了几件事：

- 把状态组织起来
- 把对象语义化
- 把可执行动作收束起来
- 把不相关自由度挡在外面
- 让人类不需要每一步都直接面对底层系统

换句话说，app 不只是界面。  
app 本身就是工作边界。

这件事放到 Agent 身上一样成立。

Agent 也不应该默认直接面对整台电脑。  
更合理的做法，是让它在更高语义的工作对象里工作。

比如：

- 文件视图
- 代码视图
- 搜索视图
- 浏览视图
- 任务视图

这时候 Agent 操作的，就不再是一个无限展开的机器环境。

而是一组更窄、更清晰、更可治理的工作对象。

所以我会把 Agent Apps 讲得更直白一点：

**Agent Apps 不是在限制 Agent 的能力。**  
**它是在重新设计 Agent 的工作边界。**

而这种“限制”不是削弱，很多时候反而是能力放大。

## 为什么工作层一旦出现，执行层和安全问题都会变得更好处理

如果 Agent Apps 真的是工作层，那它的价值就绝不只是“更安全”。

它其实会同时改善三件事。

### 1. 状态一致性会变好

很多 Agent 系统的问题，不是工具不够多，而是状态管理太差。

比如 CLI 天然就有一个问题：

它会不断追加新的输出结果。  
旧的还在，新的又来了。  
上下文越来越长，状态越来越乱。

Agent 最后不是在一个干净的环境里工作，  
而是在一堆历史观测里不断自己重建“现在到底是什么”。

而 Agent Apps 可以把状态组织成持续存在的对象：

- 文件视图可以被打开
- 文件内容可以原地更新
- 不再需要时可以移出上下文
- 搜索结果可以作为独立视图存在
- 任务状态可以持续存在，而不是不断重说一遍

一旦工作层出现，状态就不再只是“文本输出的堆积”。  
而开始变成“可以持有、更新、失效、移除”的对象。

这对可靠性非常重要。

### 2. 安全问题会更好治理

当 Agent 不是直接面对整台机器，而是在 app 级边界里工作时，安全模型会清晰很多。

因为这时候你可以把风险收束在更高语义的一层：

- 它能不能编辑这个文件视图
- 它能不能打开新的工作区
- 它能不能关闭这个对象
- 它能不能触发某个高风险动作

而不是让它默认面对：

- 任意 shell
- 任意路径
- 任意系统状态
- 任意 GUI 控件

这会让安全从“全局权限治理”变成“工作边界治理”。

后者通常更可解释，也更可控。

### 3. 效率会更高

这一点很多人一开始容易忽略。

直觉上看，好像“给 Agent 更大的自由度”会让它更强。

但很多时候，真正拖累 Agent 的，恰恰是无结构的自由度太大。

自由度太大，带来的不是纯粹的能力，而是：

- 更多错误动作空间
- 更多无效搜索空间
- 更多状态重建成本
- 更多上下文污染
- 更多对模型推理预算的浪费

所以更好的 Agent，不一定来自更大的权限。

很多时候，反而来自更好的工作层设计。

说白了：

**你不是在压制 Agent。**  
**你是在把它从一个过于原始的环境里解放出来。**

它少了很多没必要处理的噪声，反而更容易把事做对。

## 所以，本地、容器、VM 谁会赢？我的答案是：都重要，但还不够

如果回到题目本身，我的答案其实很明确：

本地、容器、VM 都会长期存在，而且都重要。

它们大概率不会形成一个“唯一正确答案”，而会按：

- 任务类型
- 风险等级
- 延迟要求
- 人机协作方式
- 治理要求

长期分工。

所以如果问题只是：

> 谁才是 Agent 时代 AI 执行层的真正主战场？

我的回答会是：

**没有单一赢家，执行层会长期分层。**

但如果把问题再往深处问一步，我的答案会变成另一句：

**真正会拉开下一阶段 Agent 系统差距的，可能不只是执行层，而是谁先把工作层做出来。**

因为：

- 本地、容器、VM 决定的是 Agent 在哪儿跑
- Agent Apps 决定的是 Agent 在什么里面工作

前者很重要。  
但后者很可能才是现在更稀缺、也更容易被低估的那层东西。

所以我现在越来越倾向于这样看：

本地、容器、VM 会长期共存。  
但真正定义下一阶段 Agent 上限的，不会只是某一种执行形态统一世界。

而是：

**谁先把执行层和工作层真正拼成一个连贯系统。**

如果这件事成立，那么 Agent Apps 就不是附属品，也不是包装层。

它会变成 Agent 时代非常基础、非常关键的一层基础设施。

大家现在都在卷 Agent 跑在哪儿。  
但下一阶段真正拼的，可能不是执行层，而是谁先把工作层做出来。
